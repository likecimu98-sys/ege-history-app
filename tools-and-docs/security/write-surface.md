# Поверхность записи Firestore → что ломают строгие правила

Выведено из фактического кода `firebase-sync.js` (2026-07-15), не из догадок.
Цель — знать заранее, кого залочит строгий ruleset, и что для этого переделать.
Кандидат правил: `security/firestore.rules.strict`. Симуляция: `security/rules-sim.mjs`.

## Установленный факт про аутентификацию

Минтинга custom-token **нет** (`createCustomToken` отсутствует во всём `/root/bot`).
`__initial_auth_token` никем не задаётся → `initAuth()` для всех уходит в
`signInAnonymously()`. Значит **`auth.uid` всегда случайный аноним, никогда не равен
docId**. Бот пишет в Firestore только Admin SDK'ом (мимо правил). Чтобы `auth.uid ==
docId`, минтинг надо ДОБАВИТЬ (см. предпосылку 1 ниже).

## Что нужно сделать ПЕРЕД включением строгих правил (по возрастанию риска)

**Предпосылка 1 — минтинг токенов (нового кода нет сейчас).**
HTTP-эндпоинт на VPS (бот уже Node + Admin SDK + serviceAccount): принять Telegram
`initData`, проверить HMAC по токену бота, вернуть
`admin.auth().createCustomToken(String(tgId), { tgId, teacher, classes })`.
Клиент на старте в Telegram зовёт его → `signInWithCustomToken`; вне Telegram — anon.
Nginx-роут + TLS + firewall. Это ЖИВАЯ инфраструктура на сервере с учениками —
делать по явному согласию, не молча.

**Предпосылка 2 — учительские мутации в бота.** Выдача/отмена ДЗ и выпуск из группы
сейчас пишутся из БРАУЗЕРА учителя в чужой `students/{uid}`. Перенести в бота
(Admin SDK) — либо новые команды бота, либо тот же эндпоинт. Тогда `allow write:
isOwner` не ломает учителя, а поверхность записи в чужие доки исчезает.

**Предпосылка 3 — слияние личностей в бота.** Клиент пишет tombstone `_mergedInto` и
сводит дубликаты в чужие доки. Это уже частично делается скриптом на VPS
(`_repair-merged.js`). Перевести штатное слияние на серверную сторону, чтобы клиент
писал только свой канонический документ.

**Предпосылка 4 — разнести данные.** `fullStateJson` (приватный прогресс) вынести из
`public/data/students/{id}` в `private/data/state/{id}`. В `students` остаются только
публичные поля лидерборда (имя, `duelRating`, `classCode`, `learnedCount`). Иначе
`read: if authed()`, нужный лидерборду/учителю, продолжает светить приватный блоб
любому анониму.

**Предпосылка 5 — Google/анон.** Google-ученик: docId `google_<uid>`, правило
`'google_'+auth.uid == studentId` (уже в кандидате). Чистые анонимы (без TG/Google)
строго-владеемыми быть не могут — либо оставить как есть, либо облачный прогресс
только после входа.

## Порядок безопасного выката

1. Предпосылки 1–4 (код бота + клиента), НИЧЕГО не деплоя в правила.
2. Экспорт копии прод-данных → Firestore-эмулятор (нужна Java) → авторитетный прогон
   `@firebase/rules-unit-testing` против `.strict` на реальной форме доков.
3. Публикация `.strict` в наименее людное время; baseline (`/firestore.rules`) в
   блокноте для мгновенного отката (в Firestore нет dry-run).
4. Наблюдение за ошибками прав в проде (сбор `permission-denied` из клиента).
