## Что это за проект
PWA-тренажёр для подготовки к ЕГЭ по истории (ваниль JS, без сборщика для JS; CSS — Tailwind
в `output.css`). Прод: **https://reshay-istoriyu.ru**. Клиент — Telegram Mini App
(@Reshay_istoriyu_bot), большинство пользователей **на мобильном**.

## Архитектура (после миграции Firebase→PostgreSQL, июль 2026)
- **Хостинг:** VPS `185.198.152.200` (root), Nginx `root /var/www/ege-app` (обычная папка),
  Let's Encrypt. `location /auth/` проксируется в API. Конфиг сайта:
  `/etc/nginx/sites-enabled/reshay-istoriyu.ru`.
- **Данные/синк:** PostgreSQL на VPS (миграция с Firebase завершена). Клиентские модули синка —
  `cloud-sync.js` + `vps-sync-compat.js`. Старый `firebase-sync.js` выведен из клиента и
  **не должен попадать в прод-архив** (`.gitattributes export-ignore`).
- **Бот:** `/root/bot/` на VPS (pm2). API/инфра: `/opt/ege-history-api`, `/opt/ege-history-infra`.
- **Кэш клиента:** Service Worker с версией `APP_VERSION` в `service-worker.js`
  (формат сейчас `2026-07-DD-vps-N`). **Меняешь кэшируемый JS/HTML/CSS — обязательно подними
  версию SW и общий `?v=RELEASE` всех стартовых JS/CSS**, иначе старый worker может собрать
  новый HTML со старыми файлами. Запись ответа в CacheStorage не должна блокировать выдачу
  сетевого ответа странице. В Telegram на iOS Service Worker намеренно отключён из-за
  зависаний WKWebView CacheStorage; обычная PWA в Safari остаётся офлайн.

## ДЕПЛОЙ — читай внимательно, тут была главная путаница
- **Рутинный деплой сайта → `powershell -File deploy-static.ps1`.**
  git archive HEAD → scp → распаковка в `/var/www/ege-app` с атомарной заменой и снимком
  отката `ege-app.prev-*`. **Базу данных не трогает.** Это правильный скрипт для обычных правок.
- **`deploy-vps.ps1` — это РАЗОВЫЙ cutover Firebase→PostgreSQL, НЕ рутинный деплой.**
  С `-Cutover -ConfirmCutover RESHAY_HISTORY_VPS` он **переключает боевую БД**, импортирует
  Firestore, ставит окна отката. **Не запускай его для правок сайта.** Cutover уже выполнен.
  Его снимки отката `ege-app.rollback-*` / `*.client-rollback-*` — 60-дневная страховка,
  их удалять/трогать нельзя.
- **SSH:** строгая проверка host key (`StrictHostKeyChecking=yes`, только `ssh-ed25519`,
  ключ `~/.ssh/id_ed25519`). Отпечаток сервера:
  `SHA256:2QvI08yVRpFJ4D35RElDDHmPBoMl88u9+qT0AhNcVKs`. **Не отключай строгую проверку и не
  делай `ssh-keyscan`** — это обесценит пиннинг.
- **Профиль Windows с пробелом** («Хозяин Саша») ломает путь `UserKnownHostsFile` в ssh.
  Оба .ps1 используют **DOS 8.3-короткие пути** (`C:\Users\90DF~1\...`). PowerShell 5.1 корёжит
  встроенные кавычки при передаче в native ssh/scp — поэтому именно короткие пути, не кавычки.
- **.ps1 — только ASCII** (PowerShell 5.1 читает скрипты в ANSI без BOM; кириллица → парс-ошибки).
- `deploy-static.ps1` после распаковки нормализует mtime файлов по часам VPS: локальные часы
  могут быть впереди сервера, а будущий `Last-Modified` ломает браузерную ревалидацию кэша.
- После деплоя: `git push origin master` (GitHub — запасной адрес/бэкап).

## Инварианты — НЕ ломать без явного согласия пользователя
- Безопасность деплоя: строгий host key + ed25519-пиннинг (см. выше).
- Прод-архив НЕ содержит `server/`, `firebase-sync.js` (гарды в обоих .ps1 — сохраняй их).
- Firestore/доступы, дневные лимиты строк, ролевые правила учителей — менять только по задаче.
- Тема: в приложении оставлена ТОЛЬКО `skin-classic` (пикер тем удалён). Другие скины в
  `theme-aurora.css` — мёртвый груз, но `body.skin-classic` активна всегда; правя чипы/цвета,
  проверяй И светлую, И тёмную тему (`html.dark`).

## Рабочее соглашение
- **Проверяй в браузере перед деплоем** (превью на localhost). Для UI — на мобильном 375px
  (целевая среда). Не выкладывай непроверенное на прод.
- Коммиты — осмысленные, с префиксом (`fix/feat/chore(область)`), на русском ок.
- Ветка по умолчанию — `master`. Прод = локальный master (может быть впереди origin, т.к.
  часть работы не в GitHub). Не делай force-push и деструктивных reset без согласия.
- Крупные/необратимые действия (переключение БД, миграции, удаление данных, смена настроек
  сервера) — только с явного «да» пользователя.
