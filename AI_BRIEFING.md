## Что это за проект

PWA-тренажёр для подготовки к ЕГЭ по истории (Россия). Ванильный JS
(без фреймворка), Telegram Mini App + обычный сайт. Прод:

ученики, есть роли «учитель»/«владелец школы».

Репозиторий: `C:\Users\Хозяин Саша\Desktop\ege-history-app-main` (Windows),
GitHub `https://github.com/likecimu98-sys/ege-history-app` (пока публичный).

## Стек и архитектура

- **Классический JS**, НЕ модульный (кроме `firebase-sync.js` — ES-модуль).
  Все файлы подключены `<script defer>` в `index.html` и вешают функции/данные
  в `window.*`. Порядок скриптов в defer-цепочке важен.
- Ключевые файлы (корень репо):
  - `data.js` (~4750 строк) — банк фактов/заданий (task1…task7 массивы).
  - `state.js` — модель состояния ученика (`window.state`, `stats`, SRS,
    achievements), `SAVE_FIELDS` — что уходит в облако.
  - `table.js` — генератор таблиц-заданий + вся анти-двусмысленная логика
    (проверки задвоения ответов по каждому типу задания).
  - `ui.js` — рендер экранов, модалки, кабинет учителя.
  - `app.js` — экшн-хендлеры кнопок (`ACTION_HANDLERS`), проверка ответов.
  - `modes.js` — карточки/флеш-карты и разные учебные режимы.
  - `firebase-sync.js` (ES-модуль, ~4000 строк) — вся синхронизация с
    Firestore, слияние состояний между устройствами, роли учителей, ДЗ,
    дуэли, лимиты, инвайты.
  - Самодостаточные оверлей-режимы: `swipe-mode.js` + `swipe-data.js`,
    `match-mode.js`, `vov-mode.js`, `visual-trainer.js` — каждый паттерн
    «IIFE + `window.openXMode`/`closeXMode` + свой z-index + свой `<style>`
    + ленивая загрузка данных инъекцией `<script>`».
  - `service-worker.js` — офлайн-кэш, версия в константе `APP_VERSION`
    (руками бампается при каждом деплое, иначе клиенты не увидят обновление).
  - `pwa.js` — регистрация SW, прогрев кэша.
  - `cram.html` — отдельная суб-страница («зубрёжка», карточки), грузится
    в iframe.
  - `tools/fipi-parser/` — отдельный локальный инструмент (Node, порт 3777):
    парсер открытого банка заданий ФИПИ + конструктор пробников (Word/PDF).
    Не часть прод-сайта, работает только локально у автора.

## Как гонять/собирать

- Стили — Tailwind: `output.css` **это сборка**, не редактируется руками.
  После правки классов в разметке/JS: `npm run build:css` (конфиг
  `tailwind.config.js`, источник `input.css`). Иначе новый класс тихо не
  применится.
- Дизайн-токены: `tokens.css` (акценты приложения) и `theme-aurora.css`
  (8 скинов — aurora/classic/constructivism/vaporwave/sakura/terminal/
  midnight/scholar, список в `SKINS` в `ui.js`).
- Нет сборщика бэкенда/бандлера для JS-логики — правишь файлы напрямую.
- Локальная проверка: обычно через встроенный браузерный препросмотр
  (открыть `index.html`/локальный сервер) и `javascript_tool`-инъекции в
  консоль для юнит-проверок генераторов таблиц (нет формального test-раннера
  для клиентского кода). Для `tools/fipi-parser` есть `*.selftest.js`.

## Деплой (ВАЖНО!)

Прод — **свой VPS**, не GitHub Pages (Pages остаётся только как запасной
адрес для старых закладок).

- Скрипт: `powershell -File deploy-vps.ps1` из корня репозитория.
  Что делает: `git archive` **последнего КОММИТА** (не рабочей копии!) →
  scp на VPS → атомарная замена `/var/www/ege-app` на сервере.
  **Значит: перед деплоем изменения обязательно закоммичены.**
- После деплоя: `git push origin master` (GitHub Pages как запасной адрес).
- **Перед каждым продовым деплоем — бампни `APP_VERSION` в
  `service-worker.js`** (например `2026-07-15-3`), иначе Service Worker
  продолжит отдавать старые файлы из кэша части пользователей.
- SSH только по ключу с ПК автора (`~/.ssh/id_ed25519`), парольного входа
  нет. **У другой модели/агента, скорее всего, НЕТ доступа к этому ключу и
  к самому VPS** — деплой на прод может выполнить только тот, у кого есть
  физический доступ к машине автора. Если ассистент не может выполнить
  `deploy-vps.ps1` — это ожидаемо, надо явно сказать пользователю, что
  изменения закоммичены/запушены, но не задеплоены на прод, и попросить
  запустить деплой самостоятельно или в сессии с доступом к VPS.

## Безопасность — известное и НЕ устранённое ограничение

Firestore правила (`firestore.rules`) сейчас: `allow read, write: if
request.auth != null` (просто «авторизован», без проверки владения
документом). Это **сознательный компромисс, не забытая дыра**:

- `request.auth.uid` ученика — случайный Firebase-uid, а не Telegram ID
  документа ученика (кастомные токены минтит бэкенд бота без привязки
  uid=tgId). Поэтому строгое правило `auth.uid == docId` **немедленно
  заблокирует всех ~200 учеников** — включать нельзя, пока бэкенд бота не
  переделан.
- Включён анонимный вход Firebase (нужен для standalone-PWA вне Telegram),
  поэтому **любой в интернете** может получить анонимный токен через
  публичный `apiKey` и читать/писать чужие документы учеников (включая
  `fullStateJson` — персональные данные несовершеннолетних, 152-ФЗ). Это
  **подтверждено живым тестом** 2026-07-15.
- Настоящий фикс — вне этого репозитория: (1) бэкенд бота минтит
  custom-token с `uid = канонический docId` (или claim `tgId`),
  (2) разнести публичные поля лидерборда и приватный `fullStateJson`,
  (3) только потом включать строгие правила.
- Подробности и вся история решений: `SECURITY.md`, `AUTH.md` в корне
  репозитория.

**Не патчить `firestore.rules` вслепую** без понимания auth.uid↔docId —
легко залочить всех пользователей.

## Жёсткие правила работы с данными/кодом (важно для любого агента)

1. **Учитель НИКОГДА не переписывает документ ученика напрямую.** Прогресс
   ученика — `fullStateJson`, перезаписывается только самим учеником при
   каждом sync. Правки со стороны учителя = гонка и затирание прогресса.
   Отмена ДЗ и т.п. — всегда через отдельные top-level поля/tombstone-записи,
   не через чтение-модификацию-запись чужого `fullStateJson`.
2. **Прямые записи в живую БД (Firestore) из сессии ассистента требуют
   явного разрешения пользователя.** Не трогать прод-базу без спроса.
3. Добавляя новое поле в `state.stats`, которое должно жить в облаке —
   регистрировать его в ЧЕТЫРЁХ местах: (1) инициализация в `state.js`,
   (2) `SAVE_FIELDS` в `state.js`, (3) merge-логика в `deepMergeStates`
   (`firebase-sync.js`), (4) `CLOUD_STATE_FIELDS` там же. Пропуск любого —
   тихая потеря данных при слиянии устройств.
4. Новый аккаунт-зависимый ключ `localStorage` → обязательно добавить в
   `IDENTITY_WIPE_KEYS` (`firebase-sync.js`), иначе смена аккаунта на общем
   ПК смешает прогресс двух разных людей.
5. **Проверка на «двойной ответ» в заданиях НЕ может опираться только на
   БД-оракул** (скрипт, сверяющий по базе данных) — он не ловит смысловые
   пересечения (например, обобщающий трейт группы против отдельного члена
   группы, или вложенный исторический процесс против общего). Нужен ещё и
   **ручной/логический просмотр сгенерированных таблиц**, когда меняется
   контент заданий. Пример пойманного вручную бага: «Тройка передвижников»
   и «Иван-царевич на сером волке» Васнецова считались двумя верными
   ответами, пока не связали через `appliesToIds`/`GROUPS` в
   `unifyTask7Applies` (`state.js`).
6. Задание 5 ЕГЭ официально включает участников СВО как экзаменационный
   материал (не позиция пользователя/автора) — строить нейтральные
   учебные инструменты под этот контент нормально, отказывать как
   «пропаганде» не нужно.
7. После деплоя SW (`service-worker.js`) **обязательно бампать
   `APP_VERSION`**, иначе часть пользователей продолжит получать старую
   версию из кэша.
8. Тяжёлые файлы данных (`visual*Data.generated.js`, ~765 КБ суммарно) НЕ
   должны попадать обратно в `defer`-цепочку `index.html` / прекэш SW —
   они специально грузятся лениво в фоне, чтобы первая загрузка (особенно
   на мобильных) не тормозила.

## Модель данных вкратце

- `state.stats` — SRS-прогресс по фактам (`factStreaks`), достижения,
  Elo дуэлей (`duelElo/duelGames/...`), `assignments[]` (ДЗ, см. ниже),
  `vovLearned{}` и т.п. per-item «выучено»-карты.
- ДЗ — **пачка (bundle) заданий**, а не одно задание:
  `assignments = [{id, title, deadline, status, items:[{task, period,
  metric:'lines'|'points'|'learned', goal, progress, done}]}]`. Метрика
  `learned` = живой счётчик выученных фактов периода на общей SRS-системе
  (не отдельный счётчик).
- Классы — просто строка `classCode` на документе ученика + отдельный
  журнал `classes/{code}.assignments` (чтобы поздно присоединившиеся
  ученики получали старые ДЗ).
- Главная кнопка лобби — `computeMainAction()` (`ui.js`): лестница
  ДЗ → ротация (новое/ошибки/повтор) → слабое место → «всё выучено».
